React performance
optimization checklist

Shivam Thakur Apr 28, 2026 7 min read

React is fast by default, but it is easy to make it slow by accident. Almost every performance problem I have debugged comes down to the same short list. Here is the checklist I run, roughly in order, whenever a React app starts feeling sluggish.

1. Measure with the Profiler first

Open the React DevTools Profiler, record an interaction, and read the flame graph. You are looking for components that render often and take a long time. Optimising without this step is guessing — and you will usually guess wrong.

2. Eliminate wasted re-renders

The number one cause of slowness is re-rendering things that did not change. Stabilise references with useMemo and useCallback, split large context providers so a change to one value does not re-render everything, and lift state only as high as it truly needs to go.

3. Memoize the expensive components

Wrap the specific components the Profiler flagged in React.memo. Do not blanket-memoize everything — memoization has a cost and only pays off when a component re-renders frequently with unchanged props.

Do not optimise what you have not measured. The Profiler tells you where the time actually goes — trust it over intuition.

4. Virtualize long lists

Rendering thousands of rows kills performance. Use a windowing library such as react-window so only the visible rows are in the DOM. This is often the single biggest win for data-heavy screens.

5. Code-split to shrink the bundle

Load routes and heavy components lazily with React.lazy and dynamic imports so the first screen ships less JavaScript. Combine with a good loading state and the app feels instant even before everything has downloaded.

FAQ

Should I wrap every component in React.memo?

No. React.memo has a cost of its own and only helps when a component re-renders often with the same props. Measure with the Profiler first, then memoize the specific components that show up as expensive and frequently re-rendered.

What is the most common cause of slow React apps?

Wasted re-renders. New object or function references created on every render, unstable context values, and large lists rendered in full are the usual culprits. Stabilise references and virtualize long lists before anything else.

Does code splitting really improve performance?

Yes, for load time. Splitting routes and heavy components with React.lazy and dynamic imports reduces the initial JavaScript bundle, so the first screen becomes interactive faster. It does not change runtime render cost, only what ships up front.
